    Historical Timeline

    The evolution of VCS spans several decades, moving from early centralized systems to modern distributed ones

    • 1982: RCS (Revision Control System) - This local VCS automated change tracking for individual files using delta storage (storing differences).
    • 1986: CVS (Concurrent Versions System) - CVS added a client-server model, allowing multiple developers to collaborate over a network. It became widely used despite limitations like non-atomic commits.
    • 2000: Subversion (SVN) - Created as a successor to CVS, SVN introduced key improvements like atomic commits (all-or-nothing changesets) and better file rename handling. It was still a centralized system.
    • 2005: Git and Mercurial - This was a breakthrough year for distributed version control (DVCS). Linus Torvalds created Git for the Linux kernel, and Mercurial was also released. In a DVCS, every user has a full copy of the repository history.

    Why Distributed VCS (DVCS) Changed the Game

    The move from Centralized (CVCS) to Distributed (DVCS) was a major shift. In a CVCS like Subversion, there is a single central repository. This model has a single point of failure; if the server is down, nobody can commit changes.

    DVCS, like Git, changed this by giving every developer their own full copy of the repository, including the entire history. This means:

    • Offline Work: Developers can commit changes locally even when offline.
    • Resilience: Every clone serves as a backup of the project.
    • Better Branching & Merging: In DVCS, merging is a "first-class operation". This made branching and merging extremely fast and easy, enabling non-linear workflows where many experimental branches are common. This flexibility is a primary reason DVCS (especially Git) became the modern standard.
